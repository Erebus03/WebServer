Webserv — Complete Project Structure

All three members · One file per class · Clear ownership boundaries · No shared mutable state

Member Overview

Member A — Network & Foundation

Owns everything that touches a file descriptor. The rest of the team builds against the interfaces A defines.

  • Config file parsing
  • Socket bind / listen / accept
  • poll() event loop
  • Client lifecycle management
  • Non-blocking read / write buffers
  • CGI pipe management in event loop
  • Connection timeout enforcement
  • Signal handling (SIGINT, SIGTERM)

Member B — HTTP & Response

Owns everything between raw bytes and clean structs — in both directions.

  • HTTP request line parser
  • Header parsing (case-insensitive map)
  • Body reading (Content-Length + chunked)
  • Multipart/form-data parser
  • HTTP response serializer
  • MIME type table
  • Default error page generator
  • CGI response header parser

Member C — Router & Handlers

Owns everything between a parsed request and a filled response. Never touches sockets directly.

  • URI → LocationConfig matching (Router)
  • Method → Handler dispatching (Dispatcher)
  • Static file serving (GetHandler)
  • File upload storage (PostHandler)
  • File deletion (DeleteHandler)
  • CGI fork / execve / env setup (CgiHandler)
  • Directory listing HTML (DirectoryLister)
  • Filesystem helpers (FileUtils)

Every Class — Owner, Job, Dependencies
Class / File Owner One job only Depends on
types.hpp shared Plain data structs only. No methods, no logic. Frozen after Day 1. Nothing — this is the root.
main.cpp shared Entry point. Parse argv, call ConfigParser, hand result to EventLoop, call run(). ConfigParser · EventLoop
ConfigParser A Read config file line by line. Populate vector<ServerConfig>. Exit with error on bad syntax. types.hpp
Socket A Wrap one listening socket: socket() → setsockopt() → bind() → listen() → fcntl(O_NONBLOCK). types.hpp
Client A Hold all state for one connection: fd, input buffer, output buffer, bytes sent, state enum, cgi_pipe_fd, cgi_pid. types.hpp
ConnectionManager A Accept new clients, remove disconnected ones, enforce timeouts, compact the pollfd array safely. Client · Socket
EventLoop A The infinite poll() loop. On POLLIN: feed bytes to HttpParser. On POLLOUT: drain output buffer via send(). Manage CGI pipe FDs. ConnectionManager · HttpParser · Dispatcher · ResponseBuilder
HttpParser B Incremental state-machine. Call with a chunk of bytes, returns INCOMPLETE or COMPLETE + populated HttpRequest. types.hpp
MultipartParser B Split a multipart/form-data body on its boundary. Return vector of parts each with headers and binary content. types.hpp
CgiResponseParser B Read CGI stdout. Split on blank line. Parse CGI headers (Status:, Content-Type:). Return HttpResponse. types.hpp
ResponseBuilder B Serialize HttpResponse → raw bytes: status line + CRLF headers + blank line + body. types.hpp · MimeTypes · ErrorPageGenerator
MimeTypes B Map file extension to Content-Type string. Static lookup table. No other logic. Nothing.
ErrorPageGenerator B Given a status code, return a minimal HTML string. Used when no custom error page is configured. Nothing.
Router C Walk ServerConfig.locations. Return the LocationConfig whose path is the longest prefix of the request URI. types.hpp
Dispatcher C Check method is allowed. Check redirect. Route to correct handler. Return 405 / 301 / handler result. Router · all handlers · types.hpp
GetHandler C Resolve URI to path. Serve file body with correct MIME type. Or trigger DirectoryLister. Return 200/403/404. FileUtils · DirectoryLister · types.hpp
PostHandler C Receive parsed multipart parts from B. Write each file to upload_dir. Return 201 Created. FileUtils · types.hpp
DeleteHandler C Resolve URI to path. Call unlink(). Return 204 on success, 404 if missing, 403 if denied. FileUtils · types.hpp
CgiHandler C fork() + execve() the CGI interpreter. Build envp. Create stdin/stdout pipes. Return pipe FD + pid to EventLoop. FileUtils · types.hpp
DirectoryLister C opendir() / readdir() a path. Build and return an HTML string listing names, sizes, dates. Nothing except POSIX.
FileUtils C resolve_path() · file_exists() · is_directory() · is_readable() · read_file(). Used by every C handler. Nothing except POSIX.

Integration Points — Where Members Meet
Integration point Who provides Who consumes What must be agreed before either starts
HttpParser → EventLoop B A The exact function signature: what B exposes, how A calls it with a byte chunk, what return value means INCOMPLETE vs COMPLETE. Agree Day 1.
EventLoop → Dispatcher C A A calls C's Dispatcher when a request is complete. Agree on the function signature: takes HttpRequest + ServerConfig reference, returns HttpResponse.
ResponseBuilder → EventLoop B A B produces a std::string of raw bytes. A puts it in Client.output_buf. No other coupling needed.
MultipartParser → PostHandler B C What does a parsed "part" look like? Agree on the Part struct (filename, content-type, data as std::string). Define it in types.hpp.
CgiHandler → EventLoop C A The hardest interface. C returns a pipe read FD and a child PID. A adds the FD to poll(), reads it, feeds output to B's CgiResponseParser. Design together in Week 3 Day 1.
CgiResponseParser → EventLoop B A A feeds accumulated CGI pipe bytes to B's parser, gets back an HttpResponse when the pipe closes (EOF).
Golden Rules — Non-Negotiable
Member A rules
1
Never call recv() or send() without poll() first confirming the FD is ready.
2
Every FD you open must be closed. Track CGI pipe FDs as carefully as socket FDs.
3
The pollfd array must never be modified while being iterated. Use a pending-add queue.
4
ConfigParser runs once. EventLoop runs forever. Keep them completely separate.
Member B rules
1
HttpParser must be incremental. Never assume a full request arrives in one recv() call.
2
HttpParser must be stateless between calls on a given Client — all state lives in the Client's parser object, not in global variables.
3
ResponseBuilder never touches a socket. It returns bytes. A puts them in the buffer.
4
Chunked decoding must be unit-tested with adversarial inputs before integration.
Member C rules
1
No handler ever touches a socket file descriptor — except CgiHandler which creates pipes.
2
Router does only string matching. Zero filesystem calls, zero HTTP logic inside Router.
3
All filesystem path resolution goes through FileUtils. Never construct raw paths inside a handler.
4
CgiHandler must not block. It forks and returns immediately. The event loop (A) handles the rest.
