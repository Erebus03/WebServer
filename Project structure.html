Member C — What each class does (one responsibility only)

Class Owner One sentence job What it reads · What it produces
Router C Finds the best matching LocationConfig for a URI using longest-prefix matching. Reads: URI string + ServerConfig. Returns: LocationConfig&
Dispatcher C Looks at the method and location config, picks the right handler, calls it, returns the response. Reads: HttpRequest + LocationConfig. Returns: HttpResponse
GetHandler C Serves a static file or triggers directory listing. Returns 404/403 if file is missing or unreadable. Reads: URI path + location root. Returns: HttpResponse with file body
PostHandler C Writes uploaded file content (parsed by Member B) to the configured upload directory. Reads: HttpRequest.body + upload dir. Returns: 201 or error response
DeleteHandler C Resolves URI to a file path and removes it. Returns 204 on success, 404 if missing. Reads: resolved file path. Returns: 204 / 404 / 403 response
CgiHandler C Forks a child process, sets env vars, pipes request body in, returns pipe FD to event loop. Reads: full HttpRequest + CGI path. Produces: pipe FD + child PID (for Member A)
FileUtils C Shared helpers used by all handlers: resolve path, check file exists, check permissions, read file. Used by all handlers internally. Not exposed outside Member C's code.
DirectoryLister C Reads a directory with opendir/readdir and generates an HTML page listing its contents. Reads: directory path. Returns: HTML string for HttpResponse.body

5 Rules That Keep This Structure Clean

1
One class, one file. Router lives only in Router.hpp and Router.cpp. Never put two classes in one file.
2
No handler knows about sockets. Handlers receive an HttpRequest and return an HttpResponse. They never touch a file descriptor directly except in CgiHandler.
3
types.hpp is read-only after Day 1. Any change to shared structs requires all three members to agree. No exceptions.
4
Each handler is testable alone. You can call GetHandler::handle(req, loc) with a fake request and a fake location config in a standalone test binary — no event loop needed.
5
FileUtils is your private toolbox. Every filesystem operation (path resolution, permission check, file read) goes through FileUtils. Never duplicate that logic inside a handler.

Member C — Build Order (what to write first)

1
Agree and freeze types.hpp with the team. Nothing else starts before this.
2
Write Router. Pure string matching. No filesystem, no HTTP. Testable immediately with a main.cpp test file.
3
Write FileUtils. Path resolution, stat, permission checks. Every handler needs this.
4
Write GetHandler. The first handler needed for Week 1 integration. Unblocks the team's first browser test.
5
Write Dispatcher. Thin glue that calls the right handler. Write after GetHandler so you have something real to dispatch to.
6
Write DeleteHandler, PostHandler, DirectoryLister in Week 2 once GET is proven stable.
7
Write CgiHandler last — in Week 3 after a design session with Member A about how the pipe FD re-enters the event loop.
